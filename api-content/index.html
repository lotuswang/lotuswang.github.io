作为研发部门，需要接收来自市场部门、其他研发部部门的需求。在对接需求的过程中，我们需要识别哪些是有效需求，哪些是无效需求，以便把精力放在做高价值的事情上，同时避免产品堆积无用的需求。在识别需求的时候，我需要了解：用户遇到了什么问题，要解决他的什么痛点，我们现在的产品是否能解决用户问题，该需求是个性化需求和公共需求，等等。 站立会议如果开得不好，团队成员没有参与感，容易流为形式，没有达到预期效果。 我们团队采用季度目标——>月度目标——>周计划的目标管理方式。团队没有严格按照敏捷的最佳实践来，我们定义为“类敏捷”开发方法。每个季度末，制定下一个季度的目标，并拆解到每个月度中。为了确保月度目标的达成，需要将月度目标有效拆解到每周的工作计划中。在没有引用站立会议前，工作时常出现偏差，加上项目执行过程出现问题在所难免，但由于成员没有有效、及时反馈，导致周总结会议时才暴露问题，进度因为没有及时纠偏，造成项目延期。 团队引用站立会议后，开始由固定成员组织，主要暴露了以下几个问题： 会议主持者偶尔无法按时主持会议，导致会议中断，就像没有养成的习惯，容易夭折 成员参与感不够，懒散、发言时随意、不能完整、清晰描述工作和问题 通过不断调整，我们逐步形成了以下会议制度和规范，站立会议也日趋成熟： 会议时间固定为每天上午9:40（公司9:30上班，为了确保参与人准时参加，且有准备时间，所以延迟10分钟开） 参会人员按照“昨天的工作完成情况、今天的计划、遇到的问题”提前准备好要讲的内容，而不是会上临时想，提升会议效率和效果 所有人必须站立，且不能做与站立会议无关的事 会议提出的问题，如果一两句话不能得出结论，由主持者记录，会后小范围讨论 会议采用周轮班制，所有人都可能成为主持者，每周由一个成员负责。
